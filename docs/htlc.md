# HTLC Atomic Swap Protocol Specification

## Status

This document describes the Hash Time-Locked Contract protocol as implemented in IntentX. It covers the cryptographic primitives, timing model, on-chain program, off-chain service, and worker. It is intended as a reference for engineers who will modify, debug, or audit the HTLC system.

---

## 1. Terminology

| Term | Definition |
|------|-----------|
| **Secret (S)** | 32-byte cryptographically random preimage. Generated by `rand::random::<[u8; 32]>()`. Known only to the initiator until claim time. |
| **Hashlock (H)** | `SHA-256(S)`. 32 bytes. Published on both chains as the lock condition. Anyone who knows S can prove knowledge of the preimage without revealing it until they choose to. |
| **Timelock (T)** | Unix timestamp after which the sender can reclaim locked funds. Source timelock T_s must be strictly greater than destination timelock T_d. |
| **Initiator** | The platform. Generates S, computes H, creates the source HTLC, and claims the destination HTLC by revealing S. |
| **Counterparty** | The solver. Observes H on the source chain, creates a mirror HTLC on the destination chain with the same H but shorter timelock. After S is revealed on-chain, uses S to unlock the source HTLC. |
| **Relayer** | The HTLC worker (`src/cross_chain/htlc/worker.rs`). Background process that drives state transitions. Not a trusted party — all its actions are verified by on-chain hash checks. |

## 2. Data Model

### 2.1 On-chain state (Solana Anchor: `programs/intentx-htlc/src/lib.rs`)

```
Account: Htlc (PDA seeds: [b"htlc", hashlock])
  sender:       Pubkey     (32B)   — who locked the funds
  receiver:     Pubkey     (32B)   — who can claim with the preimage
  mint:         Pubkey     (32B)   — SPL token mint
  hashlock:     [u8; 32]           — SHA-256(secret)
  timelock:     i64                — unix timestamp, refund allowed after this
  amount:       u64                — tokens locked
  claimed:      bool               — true after successful claim
  refunded:     bool               — true after successful refund
  bump:         u8                 — PDA bump for Htlc account
  escrow_bump:  u8                 — PDA bump for escrow token account

Account: Escrow Token Account (PDA seeds: [b"escrow", hashlock])
  authority: self (escrow_token_account itself)
  Holds the locked tokens until claim or refund.
```

**Invariant**: `claimed` and `refunded` are mutually exclusive. The Anchor program enforces:
- `claim()`: requires `!claimed && !refunded`, verifies `SHA-256(secret) == hashlock`, transfers escrow → receiver, sets `claimed = true`
- `refund()`: requires `!claimed && !refunded`, verifies `Clock::get().unix_timestamp >= timelock`, requires sender is signer, transfers escrow → sender, sets `refunded = true`

### 2.2 Off-chain state (PostgreSQL: `migrations/039_htlc.sql`)

```sql
CREATE TABLE htlc_swaps (
    id                UUID PRIMARY KEY,
    fill_id           UUID NOT NULL UNIQUE,
    intent_id         UUID NOT NULL REFERENCES intents(id),
    secret_hash       BYTEA NOT NULL,          -- H = SHA-256(S), 32 bytes
    secret            BYTEA,                   -- S, stored after create_swap + store_secret
    source_chain      TEXT NOT NULL,
    source_sender     TEXT NOT NULL,
    source_receiver   TEXT NOT NULL,
    source_token      TEXT,
    source_amount     BIGINT NOT NULL,
    source_lock_tx    TEXT,
    source_unlock_tx  TEXT,
    source_timelock   TIMESTAMPTZ NOT NULL,     -- T_s
    dest_chain        TEXT NOT NULL,
    dest_sender       TEXT NOT NULL,
    dest_receiver     TEXT NOT NULL,
    dest_token        TEXT,
    dest_amount       BIGINT NOT NULL,
    dest_lock_tx      TEXT,
    dest_claim_tx     TEXT,
    status            htlc_status NOT NULL DEFAULT 'created',
    solver_id         TEXT NOT NULL,
    error             TEXT,
    created_at        TIMESTAMPTZ NOT NULL,
    locked_at         TIMESTAMPTZ,
    claimed_at        TIMESTAMPTZ,
    completed_at      TIMESTAMPTZ
);
```

**Status enum**: `created`, `source_locked`, `dest_claimed`, `source_unlocked`, `refunded`, `expired`, `failed`

### 2.3 Cryptographic primitives (`src/cross_chain/htlc/crypto.rs`)

```rust
pub type Secret     = [u8; 32];
pub type SecretHash = [u8; 32];

pub fn generate_secret() -> Secret       { rand::random() }
pub fn hash_secret(s: &Secret) -> SecretHash {
    Sha256::new().chain_update(s).finalize().into()
}
pub fn verify_secret(s: &Secret, h: &SecretHash) -> bool {
    hash_secret(s) == *h
}
```

SHA-256 is used because:
1. It is the hash function used by the Solana Anchor program (`anchor_lang::solana_program::hash::hash`)
2. It provides 128-bit collision resistance and 256-bit preimage resistance
3. It is deterministic across all chains (same hash on EVM and Solana)

## 3. Protocol Lifecycle

### 3.1 State machine

```
         generate_secret()
         hash_secret()
              │
              ▼
          ┌────────┐     lock_funds() on source chain
          │Created │────────────────────────────────────▶┌──────────────┐
          └────┬───┘                                     │SourceLocked  │
               │                                         └──────┬───┬──┘
               │ fail_swap()                                    │   │
               ▼                                                │   │
          ┌────────┐                                            │   │
          │ Failed │◀───────────────────────────────────────────┘   │
          └────────┘   bridge fails / no route                     │
                                                                   │
                       ┌───────────────────────────────────────────┘
                       │
                       │  counterparty observes H, creates dest HTLC
                       │  record_dest_lock(dest_lock_tx)
                       │
                       │  initiator claims dest with S
                       │  record_dest_claim(secret, claim_tx)
                       ▼
                  ┌────────────┐
                  │DestClaimed │
                  └─────┬──────┘
                        │
                        │  counterparty (or anyone) reads S from dest chain
                        │  uses S to unlock source HTLC
                        │  complete_swap(unlock_tx)
                        ▼
                  ┌──────────────┐
                  │SourceUnlocked│   TERMINAL — SUCCESS
                  └──────────────┘

    Timeout path (source_timelock < NOW):
                  ┌──────────────┐
                  │SourceLocked  │──── source_timelock expired ────▶┌──────────┐
                  └──────────────┘     AND status NOT in            │ Refunded │
                  ┌────────┐           (dest_claimed,               └──────────┘
                  │Created │────────   source_unlocked)                TERMINAL
                  └────────┘
```

### 3.2 Transition table

| From | To | Guard | SQL WHERE | Method | Owner |
|------|----|-------|-----------|--------|-------|
| `created` | `source_locked` | `lock_funds()` succeeds | `status = 'created'` | `record_source_lock()` | Worker Phase 1 |
| `created` | `failed` | Bridge not found or lock fails | `status NOT IN ('source_unlocked','refunded')` | `fail_swap()` | Worker Phase 1 |
| `source_locked` | `source_locked` | Solver creates dest HTLC | `status = 'source_locked'` | `record_dest_lock()` | Worker Phase 2 |
| `source_locked` | `dest_claimed` | Initiator reveals S on dest chain | `status = 'source_locked'` | `record_dest_claim()` | Worker Phase 3 |
| `dest_claimed` | `source_unlocked` | Counterparty uses S on source chain | `status = 'dest_claimed'` | `complete_swap()` | Worker Phase 4 |
| `created`, `source_locked` | `refunded` | `source_timelock < NOW()` AND not claimed | `status IN ('created','source_locked')` | `refund_swap()` | Worker Phase 5 |

### 3.3 Verification at each transition

The service layer verifies `SHA-256(secret) == secret_hash` at three points:

1. **`store_secret()`** — before persisting S to DB (service.rs:201)
2. **`record_dest_claim()`** — before recording the claim (service.rs:291)
3. **Worker Phase 3 and Phase 4** — before submitting to the bridge (worker.rs:222 and worker.rs:314)

If any verification fails, the swap is marked `failed` and the worker logs `htlc_secret_does_not_match_hash`.

## 4. Timing Model

### 4.1 Timelock asymmetry requirement

```
T_s  = source timelock (initiator's HTLC)   — default 1800s (30 min)
T_d  = destination timelock (counterparty's) — MUST be < T_s
```

**Why T_d < T_s is required**:

Consider the sequence if T_d >= T_s:

```
t=0      Initiator locks source with T_s = 30 min
t=1      Counterparty locks dest with T_d = 30 min (WRONG: equal)
t=29min  Initiator reveals S on dest chain, claims dest funds
t=30min  Source timelock expires. Counterparty has not yet seen S
         (block confirmation delay, mempool congestion).
         Initiator refunds source HTLC.
         Result: Initiator gets BOTH source AND dest funds.
```

With T_d = T_s/2:

```
t=0      Initiator locks source with T_s = 30 min
t=1      Counterparty locks dest with T_d = 15 min
t=14min  Initiator MUST reveal S before t=15min or loses dest claim
t=14min  Initiator reveals S → now public on dest chain
t=14min  Counterparty reads S from dest chain events
t=15min  Counterparty uses S on source chain → claims source funds
t=30min  Source timelock irrelevant — already claimed
```

The gap `T_s - T_d` is the **safety margin** for the counterparty to observe the revealed secret and submit the source claim transaction. This must account for:
- Block confirmation time on both chains
- Event indexing latency
- Transaction submission and inclusion time
- Worker polling interval (5 seconds)

### 4.2 Current implementation

The source timelock is configurable per swap (`CreateHtlcParams.timelock_secs`), defaulting to 1800s. The destination timelock is the solver's responsibility — the platform tracks it via `dest_lock_tx` but does not enforce T_d < T_s in the DB.

**Consequence of misconfiguration**: If a solver sets T_d >= T_s, the initiator can claim dest funds and refund source funds. The solver loses their dest chain funds. This is the solver's risk to manage.

### 4.3 Clock sources

| Component | Clock source | Precision |
|-----------|-------------|-----------|
| Solana HTLC program | `Clock::get()?.unix_timestamp` | ~400ms slot time |
| PostgreSQL | `NOW()` / `Utc::now()` | Microsecond |
| Worker timeout check | `Utc::now()` compared to `source_timelock` column | Depends on DB-to-system clock sync |

**Risk**: If the DB clock is ahead of the chain clock, the worker may attempt a refund that the on-chain contract rejects (timelock not yet expired on-chain). The service's `refund_swap()` checks `Utc::now() < swap.source_timelock` in application code before issuing the SQL update.

## 5. Cryptographic Guarantees

### 5.1 Preimage safety

Revealing S on the destination chain is safe for the initiator because:

1. **S only unlocks HTLCs locked with H = SHA-256(S)**. No other funds are affected.
2. **The source HTLC is already locked**. The counterparty can use S to claim source funds, but this is the intended outcome — it completes the atomic swap.
3. **SHA-256 preimage resistance** (2^256 operations) means no party can derive S from H without the initiator revealing it.

### 5.2 Atomicity

The protocol achieves atomicity under these assumptions:

1. **Both chains enforce hashlock correctly**: `SHA-256(submitted_secret) == stored_hashlock`
2. **Both chains enforce timelock correctly**: Refund is only possible after the timelock
3. **The secret is the same on both chains**: Same H is used in both HTLCs
4. **T_d < T_s**: The counterparty has enough time to observe S and claim

Under these assumptions, exactly one of two outcomes occurs:

| Outcome | Initiator receives | Counterparty receives |
|---------|-------------------|----------------------|
| **Success** (S revealed before T_d) | Dest chain funds | Source chain funds |
| **Timeout** (S not revealed before T_d) | Source chain refund (after T_s) | Dest chain refund (after T_d) |

No intermediate state leaves funds stuck, provided timelocks are correctly configured and both chains are live.

### 5.3 What atomicity does NOT guarantee

- **Liveness**: If a chain halts, the counterparty may miss the reveal window
- **Fair exchange without trust**: The initiator can grief by revealing S at the last moment before T_d, leaving the counterparty minimal time to claim
- **Privacy**: S is revealed on-chain; all observers can see it

## 6. Attack Vectors

### 6.1 Front-running the claim transaction

**Attack**: Attacker observes the initiator's `claim(secret)` transaction in the mempool. Submits the same `claim(secret)` with higher gas/priority fee.

**Impact on Solana**: The Anchor program's `claim()` instruction sends tokens to `htlc.receiver`, not `msg.sender`. The attacker pays gas but the receiver still gets the tokens. The program comment at line 77 explicitly states: "Anyone can submit the claim, but tokens always go to the designated receiver."

**Impact on EVM**: Same — the claim function should be designed to send to the stored receiver, not `msg.sender`.

**Residual risk**: The attacker wastes the victim's gas. No fund loss.

### 6.2 Griefing via timeout

**Attack**: Counterparty (solver) intentionally does not create the destination HTLC. Source funds are locked until T_s expires, preventing the initiator from using those funds elsewhere.

**Impact**: Initiator's funds are locked for up to `DEFAULT_TIMELOCK_SECS` (1800s = 30 min). No fund loss — refund occurs after timelock.

**Mitigation**: The worker's Phase 5 (`phase_refund_expired`) automatically refunds expired swaps. The query:
```sql
SELECT * FROM htlc_swaps
WHERE source_timelock < NOW()
  AND status IN ('created', 'source_locked')
```

**Further mitigation**: Solver reputation tracking. Repeated timeout griefing reduces the solver's `reputation_score` and `failed_fills` count in `solvers` table. Solvers with poor reputation are deprioritized in auction ranking.

### 6.3 Incorrect secret propagation

**Attack**: Worker crashes after revealing S on the destination chain but before recording `dest_claim_tx` in the DB. On restart, the worker sees the swap as `source_locked` with `dest_lock_tx IS NOT NULL` (Phase 3 query), attempts to claim again.

**Impact**: The on-chain claim fails because `htlc.claimed == true` (Anchor program line 86: `require!(!ctx.accounts.htlc.claimed, HtlcError::AlreadyClaimed)`). No fund loss — the on-chain check is the safety net.

**DB consequence**: The swap remains `source_locked` in the DB. The secret S is now public on the destination chain. The counterparty can read S from the `FundsClaimed` event and use it to unlock the source HTLC, completing the swap. But the DB won't transition to `dest_claimed` → `source_unlocked` without manual intervention.

**Detection**: Query for swaps where `status = 'source_locked'` AND `source_timelock` is approaching AND the dest chain shows a successful claim event.

### 6.4 Race between claim and refund

**Attack**: The source timelock expires while the claim transaction is in the mempool. Both `claim(S)` and `refund()` are submitted to the source chain simultaneously.

**On-chain behavior**: The Anchor program checks both `!claimed` and `!refunded` in both instructions. Only one transaction can succeed (whichever is processed first by the validator). The other fails with `AlreadyClaimed` or `AlreadyRefunded`.

**Off-chain behavior**: The service's `refund_swap()` guards:
```rust
if swap.status == HtlcStatus::DestClaimed || swap.status == HtlcStatus::SourceUnlocked {
    return Err(HtlcError::InvalidState("Cannot refund: secret already revealed"));
}
```
And the SQL:
```sql
UPDATE htlc_swaps SET status = 'refunded' WHERE id = $1 AND status IN ('created', 'source_locked')
```
The `status IN ('created', 'source_locked')` predicate means: if the status has already been updated to `dest_claimed` (by a concurrent claim), the refund UPDATE affects 0 rows and returns `InvalidState`.

### 6.5 Counterparty uses a different hash

**Attack**: Counterparty creates the destination HTLC with H' != H (a hash they know the preimage of).

**Impact**: The initiator cannot claim the destination HTLC because the initiator's secret S satisfies H but not H'. The initiator's claim fails. After T_s, the initiator refunds the source HTLC. After T_d, the counterparty refunds the destination HTLC. No swap occurs, no fund loss.

**Detection**: The platform does not currently verify that the destination HTLC uses the same hash H. This is a trust-in-solver assumption. Adding on-chain verification would require the platform to read the destination HTLC's hashlock and compare with the stored `secret_hash`.

## 7. Implementation Mapping

### 7.1 Worker phases → Protocol steps

| Worker Phase | Function | DB query | Protocol step |
|-------------|----------|----------|---------------|
| Phase 1 | `phase_lock_source()` | `find_pending_locks()`: `status = 'created' AND source_timelock > NOW()` | Create source HTLC on-chain |
| Phase 2 | `phase_monitor_dest_lock()` | `find_awaiting_dest_lock()`: `status = 'source_locked' AND dest_lock_tx IS NULL` | Observe counterparty's dest HTLC |
| Phase 3 | `phase_claim_destination()` | `find_claimable()`: `status = 'source_locked' AND dest_lock_tx IS NOT NULL` | Reveal S on destination chain |
| Phase 4 | `phase_unlock_source()` | `find_pending_unlocks()`: `status = 'dest_claimed' AND secret IS NOT NULL` | Counterparty uses S on source chain |
| Phase 5 | `phase_refund_expired()` | `find_expired()`: `source_timelock < NOW() AND status IN ('created', 'source_locked')` | Refund expired HTLCs |

### 7.2 Anchor instructions → Service methods

| Anchor instruction | Service method | Transition |
|-------------------|---------------|------------|
| `lock_funds(hashlock, timelock, amount)` | `record_source_lock(swap_id, tx_hash)` | `created → source_locked` |
| — (solver creates mirror HTLC) | `record_dest_lock(swap_id, dest_lock_tx)` | Updates `dest_lock_tx` column |
| `claim(secret)` on dest chain | `record_dest_claim(swap_id, secret, claim_tx)` | `source_locked → dest_claimed` |
| `claim(secret)` on source chain | `complete_swap(swap_id, unlock_tx)` | `dest_claimed → source_unlocked` |
| `refund()` on source chain | `refund_swap(swap_id)` | `created\|source_locked → refunded` |

### 7.3 Anchor PDA derivation

```
HTLC account:           seeds = [b"htlc",   hashlock.as_ref()], bump
Escrow token account:   seeds = [b"escrow", hashlock.as_ref()], bump
```

The hashlock uniquely identifies each HTLC. Two swaps with different secrets but the same hash (collision) would conflict. SHA-256 collision resistance (2^128 operations) makes this infeasible.

### 7.4 Anchor events

| Event | Fields | Emitted by |
|-------|--------|-----------|
| `FundsLocked` | `htlc, sender, receiver, mint, hashlock, timelock, amount` | `lock_funds()` |
| `FundsClaimed` | `htlc, receiver, secret, amount` | `claim()` |
| `FundsRefunded` | `htlc, sender, amount` | `refund()` |

`FundsClaimed` publishes the secret in cleartext. This is the mechanism by which the counterparty learns S.

### 7.5 Secret storage lifecycle

```
create_swap()     → S generated, H = SHA-256(S) stored in DB, S returned to caller
store_secret()    → S stored in DB (htlc_swaps.secret column), verified against H
                    Worker can now retrieve S for Phase 3
Phase 3 worker    → Reads S from DB, verifies against H, submits claim(S) on dest chain
record_dest_claim → S persisted in DB alongside claim_tx
Phase 4 worker    → Reads S from DB, verifies against H, submits claim(S) on source chain
```

**Security note**: S is stored as raw bytes in PostgreSQL. In production, encrypt S with a KMS key before writing and decrypt at retrieval time. The current implementation stores plaintext for simplicity.

## 9. Invariants

These properties must hold at all times. The invariant checker (`src/chaos/verify.rs`) validates them post-chaos:

1. **INV-4a: Secret-hash binding**. For every swap where `secret IS NOT NULL`: `SHA-256(secret) == secret_hash`.
2. **INV-4b: Timelock enforcement**. For every swap where `status = 'refunded'`: `source_timelock < refund_timestamp`.
3. **INV-4c: Mutual exclusion**. No swap is both `source_unlocked` and `refunded`. This is structurally impossible with a single enum column but is enforced at the SQL level by the WHERE clauses in `complete_swap()` (`status = 'dest_claimed'`) and `refund_swap()` (`status IN ('created', 'source_locked')`).
4. **INV-4d: Terminal convergence**. Every swap past its `source_timelock` must be in a terminal state (`source_unlocked`, `refunded`, `expired`, `failed`).
